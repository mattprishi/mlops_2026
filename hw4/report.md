# Отчётик

## 1. Обработка ошибок и отказоустойчивость

Обрабатываем:

1. Невалидный или неполный вход для /predict
   - Несоответствие JSON-схеме (типы, обязательность полей) — ответ 422 (валидация FastAPI / Pydantic).
   - Отрицательные значения для целочисленных признаков (возраст, часы и т.д.) — 422 (валидаторы в PredictRequest).
   - Недостаточно данных для текущей модели: для признаков из feature_names_in_ отсутствует значение (null) — 400 с пояснением, каких фич не хватает.
   - Некорректные имена фичей относительно поддерживаемого набора — 400.

2. Ручка /updateModel
   - Пустой или заведомо некорректный run_id (после strip пустая строка) — 422 (ограничение Pydantic min_length).
   - Run или артефакт модели не найдены в MLflow — 404 при коде ошибки RESOURCE_DOES_NOT_EXIST, иначе при ошибках MLflow — 503; текущая модель в памяти не заменяется.
   - Ошибки чтения артефактов с диска / сети — 503; сервис продолжает работать со старой моделью.

3. Состояние модели
   - Модель не загружена — 503 на /predict.
   - Нефинитная вероятность на выходе пайплайна — 500 (защита от некорректного вывода).

## 2. Тесты

Добавлены pytest-тесты в каталоге tests. Предобработка: to_dataframe оставляет только нужные модели колонки и порядок, ошибки при пропусках и неизвестных именах фичей, валидация отрицательного возраста на уровне схемы. Инференс: на мок-модели проверяется диапазон вероятности 0–1 и целочисленный класс. Хэндлеры: health, predict с валидным телом и с ошибками (422 на отрицательный признак, 400 при нехватке фичей для текущей модели). updateModel с патчем load_model: успешная смена run и откат при ошибке MLflow с сохранением старой модели. Отдельный сценарий поднимает приложение с моками и прогоняет цепочку predict — updateModel — predict.
